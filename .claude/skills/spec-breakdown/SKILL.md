# 需求拆解

将以下需求拆解为可执行的任务清单：

**需求输入：** $ARGUMENTS

---

## 执行步骤

### 1. 上下文加载

读取项目文档，建立需求边界认知：

- `docs/SPEC.md` — 现有产品规格（用户场景、功能矩阵、约束边界）
- `docs/ARCHITECTURE.md` — 技术架构（分层结构、模块设计、依赖规则）
- `docs/DESIGN.md` — 视觉交互规范
- `docs/tasks/CURRENT.md` — 当前任务状态

### 2. 需求澄清

逐项检查，如有不明确的点**必须向用户提问**，不得假设：

- [ ] 目标用户是谁？（与现有 SPEC 中的用户画像一致吗？）
- [ ] 核心场景是什么？（用一句话描述用户想完成的事）
- [ ] 成功标准是什么？（怎样算"做完了"？）
- [ ] 与现有功能的关系？（扩展 / 独立 / 替代）
- [ ] 是否涉及新数据源或外部依赖？

### 3. 用户场景拆解

按 User Story 格式输出，编号延续 SPEC.md 中已有的序号：

```markdown
### US-XX: {场景名称}

**作为** {角色}，**我想** {目标}，**以便** {价值}。

**验收标准：**
1. {可验证的条件}
2. {可验证的条件}

**数据依赖：** {MCP 工具 / 本地数据 / 无}
```

### 4. 任务拆解

将每个用户场景拆解为开发任务，每个任务满足：

- **原子性** — 单个任务可独立完成和验证
- **可估量** — 明确的输入、输出、涉及文件
- **有序性** — 标注依赖关系和建议执行顺序

输出格式：

```markdown
### T1: {任务名称}
- 涉及文件: `src/...`
- 依赖: 无 / T2
- 验收: {怎样算完成}
```

### 5. 影响评估

| 维度 | 评估 |
|------|------|
| 新增文件 | 列出预计新增的文件路径 |
| 修改文件 | 列出需要修改的现有文件 |
| 架构影响 | 是否需要新增层级 / 模块 / 依赖 |
| SPEC 更新 | 需要新增的 US 编号和功能矩阵条目 |
| DESIGN 更新 | 是否涉及新的交互模式或展示格式 |

### 6. 输出产物

最终输出以下内容，等待用户确认后再执行：

1. **用户场景列表**（US-XX 格式）
2. **任务清单**（T1-TN 格式，含依赖关系）
3. **影响评估表**
4. **建议的 `docs/tasks/CURRENT.md` 更新内容**（新版本号 + 任务列表）

> 注意：本 skill 仅做分析和拆解，不执行代码编写。
