# G-Fund-Agent 架构设计

## 1. 概览

G-Fund-Agent 是一个渐进演进的智能量化交易助手，从简单的规则引擎（Phase 0）逐步演进为具备 LLM 推理能力的量化系统（Phase 4）。

**核心设计原则**：
- 渐进演进：每个 Phase 独立可用
- 风控优先：多层风控保护本金安全
- 决策可审计：关键决策由规则触发，LLM 仅辅助
- 人在回路：大额操作需人工确认

### 2.2 核心模块说明

#### 编排层 (DeepAgent)
- **职责**: 子 Agent 调度、工具编排、流程控制
- **组件**:
  - `agent.ts` - 主编排器
  - `data-fetcher.ts` - 数据获取子 Agent
  - `reporter.ts` - 报告生成子 Agent
- **特点**: LLM 驱动的智能编排

#### 数据聚合层 (Phase 1)
- **职责**: 多源数据聚合、格式归一化
- **组件**:
  - `data/context.ts` - 市场上下文聚合
  - `data/providers/` - 数据源适配器
    - `qieman.ts` - 且慢 MCP（净值数据）
    - `macro.ts` - 宏观指标（MVP）
    - `sentiment.ts` - 市场情绪（MVP）
- **特点**: 容错设计，单个数据源失败不影响整体

#### 分析引擎层 (Phase 2)
- **V1 引擎（默认，规则驱动）**:
  - `analysis-engine.ts` - 规则驱动分析引擎
  - `market-calculator.ts` - 市场计算（高点、涨跌幅、加权成本）
  - `rule-matcher.ts` - 规则匹配（档位触发、去重）
  - `portfolio-optimizer.ts` - 组合优化（建议生成、债券联动）
  - **特点**: 纯函数，确定性，可审计

- **V2 引擎（可选，策略驱动）**:
  - `analysis-engine-v2.ts` - 策略驱动分析引擎
  - 支持多策略并行执行和信号合并
  - 可选 LLM 信号增强（ENABLE_LLM_SIGNAL=true）
  - 提供策略统计和溯源信息
  - **特点**: 可扩展，支持策略版本管理

- **引擎选择**: 通过环境变量 `USE_STRATEGY_ENGINE` 切换

#### 策略层 (Phase 2)
- **职责**: 可插拔策略管理、信号合并
- **组件**:
  - `strategies/registry.ts` - 策略注册表
  - `strategies/grid-rebalance.ts` - 网格补仓策略（规则封装）
  - `strategies/llm-signal.ts` - LLM 信号增强策略
  - `strategies/version-manager.ts` - 策略版本管理
- **特点**:
  - 策略注册表支持多策略并行
  - 信号合并：规则70% + LLM 30% 加权
  - 策略版本自动存档和比较

#### 风控层 (Phase 1)
- **职责**: 多层风险检查、建议过滤
- **组件**:
  - `risk/drawdown.ts` - 回撤控制
  - `risk/concentration.ts` - 集中度检查
  - `risk/liquidity.ts` - 流动性预警
  - `risk/index.ts` - 风控聚合器
- **特点**: 三级风险提示（info/warning/error），自动阻止高风险建议

#### 数据存储层
- **状态文件** (JSON):
  - `buy-points.json` - 补仓点记录
  - `high-points.json` - 60日滚动高点
  - `triggered-tiers.json` - 已触发档位
- **外部数据源**:
  - qieman MCP - 73个工具（净值、持仓）
  - 宏观指标 API（预留）
  - 市场情绪 API（预留）

### 3.2 档位触发逻辑

```
┌─────────────────┐
│ 获取基金净值    │
└────────┬────────┘
         │
         ▼
┌─────────────────────────────┐
│ 检查60日滚动高点            │
│ ─────────────────────────  │
│ IF 超过60天未创新高:        │
│   → 重置高点为当前净值      │
│   → 清除补仓触发记录        │
│ ELIF 创新高:                │
│   → 更新高点记录            │
│ ELSE:                       │
│   → 使用现有高点            │
└────────┬────────────────────┘
         │
         ▼
┌─────────────────────────────┐
│ 计算较高点跌幅              │
│ drop = (nav - high) / high  │
└────────┬────────────────────┘
         │
         ▼
┌─────────────────────────────┐
│ 遍历补仓档位（3→2→1）       │
│ ─────────────────────────  │
│ FOR tier in [3, 2, 1]:      │
│   IF drop >= threshold:     │
│     IF tier > lastTriggered:│
│       → 触发补仓            │
│       → 记录档位            │
│       → BREAK               │
└────────┬────────────────────┘
         │
         ▼
┌─────────────────────────────┐
│ 检查止盈档位（同理）        │
│ ─────────────────────────  │
│ 基准: 加权平均补仓成本      │
│ 档位去重: tier > lastSell   │
└─────────────────────────────┘
```

## 8. 性能和可靠性

### 8.1 性能指标

| 指标 | Phase 0-1 目标 | Phase 2-4 目标 |
|------|---------------|---------------|
| 单次分析耗时 | < 30秒 | < 60秒 |
| 数据获取耗时 | < 10秒 | < 15秒 |
| 报告生成耗时 | < 5秒 | < 10秒 |
| 状态文件读写 | < 100ms | < 100ms |
| 并发用户 | 1（单用户） | 10+ |

### 8.2 可靠性设计

**容错机制**：
- 数据源失败降级（单个失败不影响整体）
- MCP 工具调用失败自动重试（最多3次）
- 状态文件损坏自动备份恢复
- 风控检查失败默认阻止建议（安全优先）

**数据一致性**：
- 状态文件原子写入（先写临时文件再重命名）
- 补仓点记录带时间戳，支持历史追溯
- 档位触发记录可回滚

**监控告警**：
- 连续失败超过3次发送告警
- 风控触发ERROR级别发送告警
- 数据源长期不可用发送告警

## 9. 安全与合规

### 9.1 数据安全

- API Key 不入库（.env + .gitignore）
- 状态文件本地存储，不上传云端
- MCP 通信使用 HTTPS + API Key 认证
- 敏感数据不记录到日志

### 9.2 操作安全

- Agent 只生成建议，不执行交易
- 大额操作需人工二次确认
- 风控层自动阻止高风险建议
- 所有决策可通过 LangSmith 追踪审计

### 9.3 合规性

- 符合个人投资者合规要求
- 不提供投资建议给他人
- 不涉及杠杆和衍生品
- Phase 4 执行自动化需券商API授权
