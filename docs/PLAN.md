# G-Fund-Agent 开发规划

## 产品定位

面向个人投资者的智能量化交易助手。通过 LLM Agent 实现策略开发、回测验证、信号生成和自动化执行的闭环。

## 演进路线

```
Phase 0 (当前)        Phase 1           Phase 2           Phase 3          Phase 4
稳定 V1              数据+风控          策略抽象层         回测+自优化       执行自动化
─────────────────────────────────────────────────────────────────────────────────────
固定规则引擎    →   多源数据+风控层  →  自然语言→策略  →  历史回测验证  →  交易API对接
单一MCP数据     →   宏观/行业/情绪   →  策略注册表     →  归因分析      →  订单管理
Markdown报告    →   风险预警         →  LLM信号生成    →  参数自动调优  →  执行确认闭环
手动触发执行    →   定时触发         →  策略版本管理   →  策略对比评分  →  异常自动熔断
```

## Phase 1：数据丰富 + 风控层（当前阶段）

> 目标：从 "只看净值涨跌" 升级为 "多维度市场感知 + 风险约束"

### 数据层扩展

- [ ] **接入更多数据源**
  - 宏观指标：十年期国债收益率、社融数据、PMI（判断市场周期）
  - 行业指数：中证行业指数（判断行业轮动）
  - 市场情绪：融资融券余额、北向资金、基金申赎比
  - 实现：新增 MCP 服务或直接 API 调用，统一到 `src/data/` 层

- [ ] **数据归一化层**
  - 不同数据源返回格式不同，需要统一的 `MarketContext` 类型
  - 包含：净值数据 + 宏观环境 + 行业动态 + 情绪指标

### 风控机制

- [ ] **最大回撤控制**
  - 单只基金回撤超过阈值时强制止损（当前没有止损逻辑）
  - 总组合回撤预警

- [ ] **集中度限制**
  - 单只基金不得超过总仓位的 X%
  - 单一行业不得超过 Y%

- [ ] **流动性检查**
  - QDII 基金赎回 T+7，补仓时需考虑资金到位时间
  - 大额赎回预警

- [ ] **定时触发**
  - 支持 cron 定时运行（每两周周四 / 每月 13 号）
  - 替代当前的手动触发模式

## Phase 2：策略抽象层

> 目标：从 "一套固定规则" 升级为 "可插拔的策略体系 + LLM 辅助信号生成"
>
> 核心原则演进：从 "禁止 LLM 判断" → "LLM 在风控护栏内辅助判断"

### 策略框架

- [ ] **策略注册表**
  - 每个策略是一个独立模块：名称、描述、入参、信号生成函数、风控约束
  - 当前的 `rules-config.ts` 变为第一个内置策略 "网格补仓策略"
  - 支持同时运行多个策略，按权重合并信号

- [ ] **自然语言 → 策略定义**
  - 用户用自然语言描述意图，LLM 生成策略配置
  - 示例："当沪深300从高点跌 10% 时，分三次抄底，每次加仓 5000 元"
  - LLM 输出结构化的策略 JSON，人工确认后注册

- [ ] **策略版本管理**
  - 每次修改策略参数自动存档
  - 可以回滚到历史版本

### LLM 信号增强

- [ ] **市场研判 Agent**
  - 输入：MarketContext（Phase 1 的多源数据）
  - 输出：市场环境判断（牛市/熊市/震荡）、行业机会提示
  - 约束：只提供信号建议，最终决策仍需通过风控层

- [ ] **信号合并机制**
  - 规则信号（确定性）+ LLM 信号（概率性）
  - 只有两者一致时才执行
  - LLM 信号可以调整仓位比例，但不能突破风控限制

### 架构变化

```
src/strategies/
├── registry.ts            # 策略注册表
├── grid-rebalance.ts      # 内置策略：网格补仓（当前逻辑）
├── trend-follow.ts        # 未来：趋势跟踪策略
└── llm-signal.ts          # LLM 辅助信号

核心原则：
- 规则信号 = 必要条件（通过才可执行）
- LLM 信号 = 增强条件（可调整幅度，不可突破规则）
- 风控层 = 否决权（任何信号都必须通过风控）
```

## Phase 3：回测 + 自优化

> 目标：策略不再靠拍脑袋定参数，用历史数据验证 + 自动调优

- [ ] **历史数据存储**
  - 将 JSON 文件迁移到 SQLite（净值历史、操作记录、策略快照）
  - 保留至少 3 年净值数据用于回测

- [ ] **回测引擎**
  - 输入：策略配置 + 历史数据区间
  - 输出：年化收益、最大回撤、夏普比率、胜率
  - 支持对比多个策略或同策略不同参数

- [ ] **参数自动调优**
  - LLM 分析回测结果，建议参数调整方向
  - 网格搜索 + LLM 启发式结合，找到更优参数组合
  - 人工确认后应用

- [ ] **策略评分看板**
  - 每个策略的实盘表现 vs 回测表现
  - 策略衰减预警（实盘持续跑输回测）

## Phase 4：执行自动化

> 目标：从 "看报告手动操作" 到 "确认后自动执行"

- [ ] **交易 API 对接**
  - 通过且慢或券商 API 提交交易指令
  - 严格的二次确认机制（Agent 生成指令 → 用户确认 → 执行）

- [ ] **订单管理**
  - 跟踪订单状态：已提交 / 确认中 / 已成交 / 失败
  - 成交后自动更新 portfolio.md 和 buy-points.json

- [ ] **异常熔断**
  - 单日操作金额超限自动暂停
  - 市场极端波动（如单日跌幅 > 5%）自动暂停策略
  - 连续亏损自动降低仓位

## 架构演进全景

```
Phase 0 (当前)                    Phase 2+ (目标)
────────────────                  ────────────────

用户手动触发                       定时/事件触发
     │                                 │
data_fetcher (MCP)                数据聚合层 (MCP + API + 爬虫)
     │                                 │
analyze_portfolio (纯规则)         策略引擎 (规则 + LLM 信号)
     │                                 │
                                  风控层 (回撤/集中度/流动性)
     │                                 │
reporter (Markdown)               执行层 (API交易 + 订单管理)
     │                                 │
人工操作                           人工确认 → 自动执行
```

## 关键设计决策

1. **渐进演进** — 每个 Phase 独立可用，不依赖后续 Phase 才能工作
2. **风控优先** — Phase 1 先补风控，再谈 LLM 信号和自动执行
3. **LLM 角色演进** — 从 "禁止判断" → "辅助判断（风控护栏内）" → "策略建议（人工确认）"
4. **人在回路** — 即使到 Phase 4，大额操作仍需人工确认，熔断机制不可覆盖
5. **补仓金额基于类别目标** — `类别目标金额 × 档位百分比`（如宽基第 1 档 = 200,000 × 35% × 5% = 3,500 元）
